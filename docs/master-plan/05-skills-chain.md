## 8.1 Tracker Adapter Pattern

This block is included in every skill that needs tracker access:

```
## Tracker Resolution

Read `spec-os/config.yaml`. Identify current repo name from git remote or directory.
Match against tracker.repos[].name. Use that entry's type.

If type: ado
  Use Azure DevOps MCP for all tracker operations.
  get-feature    -> mcp__ado__get_work_item(id)
  create-us      -> mcp__ado__create_work_item(type, fields)
  update-status  -> mcp__ado__update_work_item(id, fields)
  create-pr      -> mcp__ado__create_pull_request(...)
  get-capacity   -> mcp__ado__get_sprint_capacity(...)

If type: github
  Use GitHub MCP for all tracker operations.
  get-feature    -> mcp__github__get_issue(id)
  create-us      -> mcp__github__create_issue(fields)
  update-status  -> mcp__github__update_issue(id, state, labels)
  create-pr      -> mcp__github__create_pull_request(...)

If no match: stop. Report "No tracker configured for repo {name} in spec-os/config.yaml"
```

## 8.2 Context Loading (3-Tier Pattern)

This block is included in `/spec-os-implement`:

```
## Context Loading

Read task definition. Check context-level field.

TIER 1 (always):
  Read: CLAUDE.md (first 20 lines only)
  Read: tasks.md (target task section only)
  Read: spec.md (frontmatter only)

TIER 2 (context-level: 2 or higher — default):
  Invoke /spec-os-inject with task keywords and subagent type
  Read: spec-os/specs/{domain from spec frontmatter}/spec.md (if exists)
  Read: session-log.md (entry for depends-on task only)

TIER 3 (context-level: 3):
  Read: spec.md (full, or sections matching task scope)
  Read: spec-os/specs/knowledge-base.md (filtered by task domain and stack tags)
  Read: relevant ADRs from docs/design/adr/ if referenced in spec.md
```

## 8.4 Full Skill Chain

```
[PORTFOLIO ENTRY POINT — Claude Desktop personal skill]
  /spec-os-explore -> analyzes cross-repo business initiative
                   -> optionally creates Epic in ADO (dev confirms)
                   -> generates: {timestamp}-initiative-{slug}.md (master)
                                 {timestamp}-initiative-{slug}-{app}.md (one per affected app)
                   -> notifies dev to run /spec-os-brainstorm per app with context package

[PROJECT SETUP — run once per project, in this order]
  /spec-os-product -> creates docs/ (mission.md, roadmap.md, design/ structure)
                   -> modes: Initialize | Generate (from codebase) | Update
                   -> notifies dev to run /spec-os-init next

  /spec-os-init -> reads docs/mission.md optionally (seeds config.yaml + CLAUDE.md description)
               -> creates spec-os/ + .claude/ structure
               -> instructs dev (no handoff)

[3 PROJECT ENTRY POINTS — per repo]
  /spec-os-brainstorm -> receives optional {timestamp}-initiative-{slug}-{app}.md as input
                      -> resolves Feature in tracker (search existing | create new — dev confirms)
                      -> creates origin.md (with feature-id, complexity)
                      -> notifies dev [HANDOFF to /spec-os-create — dev confirms]
  /spec-os-bug:
    SIMPLE: creates changes/B{ID}-{name}/ with minimal spec.md + tasks.md (T01 only)
            -> notifies dev [HANDOFF to /spec-os-implement — dev confirms]
    COMPLEX: creates analysis.md -> notifies dev to review
             dev reviews -> invokes /spec-os-create manually -> /spec-os-plan -> /spec-os-implement
  /spec-os-init -> creates full spec-os structure + domain specs -> instructs dev (no handoff)

/spec-os-create
  reads origin.md (or asks for tracker ID)
  reads tracker Feature via adapter
  reads docs/design/00-overview.md + domain file
  validates docs/design/ exists (guard — stops if not found, instructs dev to run /spec-os-product first)
  validates domain exists in spec-os/specs/_index.md (guard — stops if not found)
  conflict check against spec-os/changes/
  creates: spec.md + spec-delta.md (empty) + session-log.md (empty) + verify-report.md (empty)
  origin.md: preserved as-is
  -> /spec-os-plan [HANDOFF — dev confirms]

  UPDATE MODE (invoked by /spec-os-implement during RECONCILE):
  reads current spec.md + proposed change
  proposes formal delta entry
  dev approves
  writes spec.md update + spec-delta.md entry
  returns list of tasks affected

/spec-os-plan
  reads spec.md
  defines US with AC (Given/When/Then + edge cases)
  applies sizing rules
  creates US in tracker via adapter
  generates tasks.md (with context-level + ux-impact + claimed-by + lessons-pending per task)
  -> /spec-os-implement T01 [HANDOFF — dev confirms]

  UPDATE MODE (invoked by /spec-os-implement during RECONCILE):
  reads tasks.md + affected tasks from /spec-os-create Update
  proposes adjustments (scope, done-when, new tasks, US split)
  dev approves
  updates tasks.md + tracker if US split

/spec-os-implement [one session per task]
  INIT: resolve tracker config
        check claimed-by: if already claimed by another dev -> stop
        assign claimed-by: {current dev}
        check task status: if in-progress -> RESUME MODE
          load session-log.md entry for this task as additional context
          continue from last known state
  LOAD: context by tier (1 always; 2-3 per context-level)
  STANDARDS: invoke /spec-os-inject with task keywords + stack from spec.md frontmatter
  AGENT: read spec-os/agents/{backend-dev | frontend-dev}.md
         build subagent context: agent identity + injected standards
  CONFIRM: identify agent type -> notify dev -> wait for approval [HANDOFF]
  EXECUTE: invoke Agent tool with combined context (isolated subagent context)
  RECONCILE: after execution, two checks:
    1. git diff -- spec.md: if changed by dev -> invoke /spec-os-create Update [HANDOFF — dev confirms]
    2. reasoning check: "Did implementation reveal anything undocumented in spec.md?"
       if yes -> propose /spec-os-create Update [HANDOFF — dev confirms]
       if tasks affected by spec change -> invoke /spec-os-plan Update [HANDOFF — dev confirms]
  COMMIT: atomic commit
  UPDATE: tasks.md status (in-progress -> done) + lessons-pending in task
  LOG: write session-log.md entry
  IF US complete:
    -> /spec-os-verify       [HANDOFF — dev confirms]
    PASS:
      -> /spec-os-doc        [HANDOFF — dev confirms] (if any task has ux-impact: true)
      -> /spec-os-sync       [HANDOFF — dev confirms] (always last)
    FAIL:
      -> notify dev -> /spec-os-implement (/spec-os-sync does NOT run)
  ELSE:
    notify dev of next task

/spec-os-sync [per US completion — always last]
  extract lessons with pending: true from session-log
  validate domain tags against spec-os/specs/_index.md
  update spec-os/specs/knowledge-base.md (with tags)
  check archival: only if dev confirms Feature done in tracker

/spec-os-verify [per US]
  reads: DoD from spec.md, AC from tracker, tasks done-when, spec-delta.md
  executes: AC coverage, tests (local), scope, conventions, DoD
  writes: verify-report.md entry
  PASS:
    create PR using pr-template.md
    update tracker US to in-review
    notify dev -> wait for approval before /spec-os-doc [HANDOFF]
  FAIL simple: notify dev -> wait for approval -> /spec-os-implement [HANDOFF]
  FAIL complex: deep diagnosis -> enriched context -> notify dev -> wait -> /spec-os-implement [HANDOFF]

/spec-os-doc [conditional — only if ux-impact]
  reads US from tracker
  updates docs/manual/{domain}.md
  -> /spec-os-sync [HANDOFF — dev confirms]
```

## 8.1 Tracker Adapter Pattern

This block is included in every skill that needs tracker access:

```
## Tracker Resolution

Read `spec-os/config.yaml`. Identify current repo name from git remote or directory.
Match against tracker.repos[].name. Use that entry's type.

If type: ado
  Use Azure DevOps MCP for all tracker operations.
  get-feature    -> mcp__ado__get_work_item(id)
  create-us      -> mcp__ado__create_work_item(type, fields)
  update-status  -> mcp__ado__update_work_item(id, fields)
  create-pr      -> mcp__ado__create_pull_request(...)
  get-capacity   -> mcp__ado__get_sprint_capacity(...)

If type: github
  Use GitHub MCP for all tracker operations.
  get-feature    -> mcp__github__get_issue(id)
  create-us      -> mcp__github__create_issue(fields)
  update-status  -> mcp__github__update_issue(id, state, labels)
  create-pr      -> mcp__github__create_pull_request(...)

If no match: stop. Report "No tracker configured for repo {name} in spec-os/config.yaml"
```

## 8.2 Context Loading (3-Tier Pattern)

This block is included in `/spec-os-implement`:

```
## Context Loading

Read task definition. Check context-level field.

TIER 1 (always):
  Read: CLAUDE.md (first 20 lines only)
  Read: tasks.md (target task section only)
  Read: spec.md (frontmatter only)

TIER 2 (context-level: 2 or higher ŌĆö default):
  Invoke /spec-os-inject with task keywords and subagent type
  Read: spec-os/specs/{domain from spec frontmatter}/spec.md (if exists)
  Read: session-log.md (entry for depends-on task only)

TIER 3 (context-level: 3):
  Read: spec.md (full, or sections matching task scope)
  Read: spec-os/specs/knowledge-base.md (filtered by task domain and stack tags)
  Read: relevant ADRs from docs/design/adr/ if referenced in spec.md
```

## 8.4 Full Skill Chain

```
[PORTFOLIO ENTRY POINT ŌĆö Claude Desktop personal skill]
  /spec-os-explore -> analyzes cross-repo business initiative
                   -> optionally creates Epic in ADO (dev confirms)
                   -> generates: {timestamp}-initiative-{slug}.md (master)
                                 {timestamp}-initiative-{slug}-{app}.md (one per affected app)
                   -> notifies dev to run /spec-os-brainstorm per app with context package

[PROJECT SETUP ŌĆö run once per project, in this order]
  /spec-os-product -> creates docs/ (mission.md, roadmap.md, design/ structure)
                   -> modes: Initialize | Generate (from codebase) | Update
                   -> notifies dev to run /spec-os-init next

  /spec-os-init -> reads docs/mission.md optionally (seeds config.yaml + CLAUDE.md description)

                      -> notifies dev [HANDOFF to /spec-os-design ŌĆö dev confirms]

    SIMPLE: creates changes/B{ID}-{name}/ with minimal spec.md + tasks.md (T01 only)

  proposes adjustments (scope, done-when, new tasks, US split)
  dev approves
  updates tasks.md + tracker if US split
```
